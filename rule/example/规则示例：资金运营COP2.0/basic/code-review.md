# 代码审查指南

## 代码审查维度
代码审查应围绕以下核心维度展开，每个维度都有其独特的检查重点和评估标准。

### 1. 系统资源管理
**关注点**: 确保所有系统资源得到正确管理和释放

**关键检查项**:
- [ ] 文件流、网络连接、数据库连接的正确关闭
- [ ] 线程池的生命周期管理
- [ ] 缓存策略的合理性设计
- [ ] 内存对象的及时回收机制
- [ ] 外部进程的清理和回收

**具体审查要点**:
- [ ] 未关闭的IO资源（FileInputStream、FileOutputStream、BufferedReader等）
- [ ] 缓存未设置失效策略或容量限制
- [ ] 非静态内部类隐式持有外部类引用导致内存泄漏
- [ ] 大对象未池化复用（如ByteBuffer、SimpleDateFormat、Calendar）
- [ ] 软引用/弱引用误用导致对象被过早回收
- [ ] 集合预估容量未初始化导致频繁扩容
- [ ] ThreadLocal使用后未清理导致内存泄漏
- [ ] 使用Runtime.exec()启动外部进程未回收

**评估标准**: 资源使用是否遵循"申请-使用-释放"的完整生命周期

### 2. 多线程并发
**关注点**: 并发环境下的数据一致性和协作安全

**关键检查项**:
- [ ] 共享状态的访问控制策略
- [ ] 同步原语的选择和使用
- [ ] 线程间数据传递的安全性
- [ ] 竞争条件的识别和解决
- [ ] 并发算法的正确性验证

**具体审查要点**:
- [ ] 双重检查锁未使用volatile修饰导致重排序问题
- [ ] ConcurrentModificationException风险（遍历集合时修改）
- [ ] ForkJoinPool存在资源竞争风险
- [ ] 锁顺序不一致导致死锁
- [ ] 可变共享对象未封装为线程安全结构
- [ ] synchronized锁住非共享对象（如局部变量）
- [ ] 原子操作与volatile的误用
- [ ] 线程池参数配置不当（核心线程数、最大线程数、队列容量）

**评估标准**: 多线程场景下系统的行为确定性和数据一致性

### 3. 容错与恢复
**关注点**: 系统在异常情况下的稳定性和自愈能力

**关键检查项**:
- [ ] 错误检测和分类的完整性
- [ ] 异常传播路径的合理性
- [ ] 故障隔离和影响范围控制
- [ ] 自动恢复机制的有效性
- [ ] 降级策略的可行性

**具体审查要点**:
- [ ] 异常捕获过于宽泛（catch Exception）
- [ ] 异常被吞掉未记录或处理
- [ ] finally块中存在return语句
- [ ] 异常处理中抛出新的异常导致原始异常丢失
- [ ] 未对受检异常进行适当处理
- [ ] 异常信息泄露敏感信息
- [ ] 重试机制缺少退避策略
- [ ] 熔断器模式实现不当
- [ ] 超时设置不合理导致资源浪费

**评估标准**: 异常情况下的系统可用性和用户体验

### 4. 数据完整性
**关注点**: 数据的准确性、一致性和可靠性

**关键检查项**:
- [ ] 数据验证的完整性
- [ ] 事务边界的设计合理性
- [ ] 数据转换的准确性
- [ ] 并发修改的冲突处理
- [ ] 数据备份和恢复机制

**具体审查要点**:
- [ ] equals/hashCode未成对重写导致集合操作异常
- [ ] 违反方法契约（compareTo、equals的对称性、传递性）
- [ ] 使用已废弃的API（如Date、Vector、Hashtable）
- [ ] 正则表达式编译逻辑错误或性能问题
- [ ] Optional滥用或链式调用空指针风险
- [ ] 正则贪婪匹配造成歧义
- [ ] 浮点数精度问题（使用BigDecimal处理金额）
- [ ] 字符串拼接使用+操作符导致性能问题

**评估标准**: 数据操作的正确性和一致性

### 5. 系统处理性能
**关注点**: 系统处理能力和资源利用的最优化

**关键检查项**:
- [ ] 计算复杂度的合理性分析
- [ ] 数据访问模式的优化
- [ ] 缓存命中率的提升策略
- [ ] 对象实例化的频率控制
- [ ] 内存分配和回收的优化

**具体审查要点**:
- [ ] 循环体中频繁创建对象
- [ ] 魔术数字未常量化
- [ ] 不必要的装箱拆箱操作
- [ ] 字符串比较使用==而非equals
- [ ] 集合遍历使用传统for循环而非增强for循环
- [ ] 数据库查询N+1问题
- [ ] 大量小对象创建导致GC压力
- [ ] 反射调用频繁使用影响性能
- [ ] 日志级别设置不当导致性能问题

**评估标准**: 系统性能指标是否达到预期目标

### 6. 安全漏洞防护
**关注点**: 系统安全性和敏感数据保护

**关键检查项**:
- [ ] 输入数据的验证和过滤
- [ ] 敏感信息的加密处理
- [ ] 权限控制的完整性
- [ ] 安全漏洞的预防
- [ ] 日志记录的安全性

**具体审查要点**:
- [ ] 路径遍历漏洞（Path Traversal）
- [ ] 不安全的随机数生成（使用Random而非SecureRandom）
- [ ] JWT签名/验证实现缺陷
- [ ] 反射调用暴露内部方法
- [ ] XML外部实体注入（XXE）
- [ ] 日志中泄露敏感信息（Log Injection）
- [ ] 未设置HTTP请求超时参数
- [ ] SQL注入风险（使用字符串拼接构建SQL）
- [ ] 密码明文存储或弱加密
- [ ] 跨站脚本攻击（XSS）风险

**评估标准**: 安全风险是否得到有效控制

### 7. 代码健壮性
**关注点**: 代码的稳定性和可维护性

**关键检查项**:
- [ ] 边界条件的处理完整性
- [ ] 参数验证的严格性
- [ ] 错误处理的全面性
- [ ] 代码的可读性和可理解性
- [ ] 扩展性和可维护性设计

**具体审查要点**:
- [ ] switch语句缺失default分支
- [ ] 枚举未穷尽处理，未来新增可能异常
- [ ] toString()实现缺失关键字段
- [ ] 方法参数未进行null检查
- [ ] 数组越界风险
- [ ] 除零操作未检查
- [ ] 类型转换未进行instanceof检查
- [ ] 方法过长，职责不清晰
- [ ] 类职责过多，违反单一职责原则

**评估标准**: 代码在各种场景下的稳定性和适应性

### 8. 依赖风险分析
**关注点**: 外部依赖的安全性和稳定性

**关键检查项**:
- [ ] 依赖版本的兼容性分析
- [ ] 传递依赖的显式声明
- [ ] 安全漏洞的及时修复
- [ ] 冗余依赖的清理
- [ ] 依赖更新的风险评估

**具体审查要点**:
- [ ] 依赖库版本冲突
- [ ] 未显式声明的传递依赖
- [ ] 引入存在CVE漏洞的组件
- [ ] 无用冗余依赖未清理
- [ ] 快照版本依赖不稳定
- [ ] 未使用的依赖未剔除造成构建臃肿
- [ ] Maven/Gradle配置不当
- [ ] 依赖范围设置错误（compile、provided、test）
- [ ] 循环依赖问题

**评估标准**: 依赖关系的稳定性和安全性

## 代码审查流程

### 阶段一：预审查准备
**目标**: 建立审查上下文，明确审查重点

**执行步骤**:
1. **变更理解**: 深入理解代码变更的业务背景和技术目标
2. **影响分析**: 评估变更对系统其他部分的影响范围
3. **风险识别**: 基于变更内容识别潜在的高风险区域
4. **审查计划**: 制定针对性的审查策略和重点检查项

### 阶段二：系统性审查
**目标**: 全面检查代码质量，发现潜在问题

**执行步骤**:
1. **架构层面**: 检查整体设计的一致性和合理性
2. **模块层面**: 验证模块间的接口设计和依赖关系
3. **方法层面**: 审查具体实现的逻辑正确性
4. **细节层面**: 检查代码细节和边界条件处理

### 阶段三：问题分析与建议
**目标**: 对发现的问题进行深入分析并提供改进建议

**执行步骤**:
1. **问题分类**: 按照影响程度和紧急程度对问题进行分类
2. **根因分析**: 深入分析问题的根本原因
3. **解决方案**: 提供具体的修复建议和最佳实践
4. **预防措施**: 建议预防类似问题的机制和流程

## 问题分级体系

| 风险等级 | 特征描述 | 处理策略 | 典型场景 |
|----------|----------|----------|----------|
| 严重性问题 (Critical) | 可能导致系统崩溃、数据丢失或安全漏洞 | 必须立即修复，不允许合并到主分支 | 内存泄漏、空指针异常、SQL注入漏洞 |
| 重要性问题 (High) | 可能影响系统稳定性或用户体验 | 在下一个版本中必须修复 | 性能瓶颈、资源竞争、异常处理不当 |
| 一般性问题 (Medium) | 可能影响代码可维护性或存在潜在风险 | 在合理时间内安排修复 | 代码重复、命名不规范、缺少注释 |
| 改进建议 (Low) | 代码质量提升建议，不影响功能正确性 | 在后续重构中考虑改进 | 代码风格优化、性能微调、架构改进 |

## 审查技巧与策略

### 审查视角切换
- **用户视角**: 从最终用户的角度审视功能实现
- **维护者视角**: 从代码维护的角度评估可读性和可维护性
- **测试者视角**: 从测试的角度思考边界条件和异常场景
- **架构师视角**: 从系统架构的角度评估设计合理性

### 审查重点识别
- **变更密集区域**: 重点关注修改频繁的代码区域
- **复杂逻辑区域**: 特别关注业务逻辑复杂的部分
- **外部接口区域**: 仔细检查与外部系统的交互代码
- **性能敏感区域**: 重点关注可能影响性能的代码段

### 问题发现技巧
- **模式识别**: 识别常见的反模式和问题模式
- **边界分析**: 检查边界条件和异常情况的处理
- **依赖追踪**: 追踪数据流和控制流的完整性
- **一致性检查**: 验证命名、风格、设计的一致性

## AI审查指导

### AI审查原则
- **系统性检查**: 按照8个核心维度进行系统性审查
- **优先级导向**: 重点关注严重性和重要性问题
- **具体化建议**: 提供具体的修复建议和代码示例
- **上下文感知**: 结合项目技术栈和业务场景进行分析

### AI审查流程
1. **代码理解**: 分析代码结构、业务逻辑和技术实现
2. **维度检查**: 按照8个维度逐一检查潜在问题
3. **问题分类**: 根据严重程度对发现的问题进行分类
4. **建议生成**: 提供具体的修复建议和最佳实践
5. **总结输出**: 生成结构化的审查报告
