## 1 布局结构规范
1.1 一般规则
【规则1-1-1】每个源程序文件的头部必须包含文件头部说明和修改记录（与头部说明合
并）。
【规则1-1-2】每个函数头部必须包含函数头部说明。
【建议1-1-1】每个源程序文件的尾部最好有文件结束说明或标志。
1.2 文件头部说明
【规则1-2-1】源程序的文件头部说明必须包含系统名称、模块名称、文件名称、软件版
权、功能说明、系统版本、开发人员、开发时间和修改记录等几个部分，审核人员、审核日
期和相关文档为可选项，各部分次序参见《总则》说明；


【规则1-2-2】源程序的修改记录说明必须包含修改日期、修改人员、修改说明，每次修
## 改为一条记录。
1.3 函数头部说明
【规则1-3-1】函数的头部说明必须包含函数名称、函数功能、输入参数、输出参数、返
回值等几个部分，对环境引用、对环境影响、内部变量说明、内部变量说明和被调函数列表
为可选项，各部分次序如下：

## 2 书写排版规范
2.1 一般规则
【规则2-1-1】不允许把多个短语句写在一行中，即一行最多只写一条语句。
【规则2-1-2】合理使用空格、空行、对齐、缩行、长行拆分等方式格式化源代码，使程
序舒展，可读。
2.2 空格
【规则2-2-1】关键字之后要留空格。象const、define、case 等关键字之后至少要留一个
空格，否则无法辨析关键字。象if、for、while、switch 等关键字之后应留一个空格再跟左
括号‘（’，以突出关键字。
【规则2-2-2】‘，’之后要至少要留一个空格；
【规则2-2-3】如果‘;’不是一行的结束符号，其后要留一个空格。
【规则2-2-4】赋值操作符、比较操作符、算术操作符、逻辑操作符、位域操作符，如“=”、
“+=” “>=”、“<=”、“+”、“*”、“%”、“&&”、“||”、“<<”,“^”等二元操作符的前后
应当各加一个空格。
【规则2-2-5】注释前后适当加一个或若干个空格。
【规则2-2-6】函数名之后紧跟左括号‘（’，中间不要留空格，以与关键字区别。
【规则2-2-7】‘（’之后和‘）’、‘，’、‘;’之前都不留空格。
【规则2-2-8】一元操作符如“!”、“~”、“++”、“--”、“*” （指针取值符）、“&”（地址
运算符）等前后不得加空格。
【规则2-2-9】象“［］”、“.”、“->”这类操作符前后不加空格。

## 3 命名规范
3.1 一般规则
【规则3-1-1】命名应当直观且可以拼读，可望文知意，便于记忆和阅读。
标识符最好采用英文单词或其组合，避免使用拼音。程序中的英文单词一般不会太复杂，
用词应当准确。例如不要把CurrentValue 写成NowValue。
【规则3-1-2】命名的长度应当符合“min-length && max-information”原则。
C 是一种简洁的语言, 命名也应该是简洁的。例如变量名maxval 就比
maxValueUntilOverflow 好用。标识符的长度一般不要过长，较长的单词可通过去掉“元音”
形成缩写。
【建议3-1-1】尽量避免名字中出现数字编号，如Value1,Value2 等，除非逻辑上的确需
要编号。这是为了防止程序员偷懒，不肯为命名动脑筋而导致产生无意义的名字（因为用数
字编号最省事）。
3.2 标示符命名
【规则3-2-1】程序中不得出现仅靠大小写区分的相似的标识符。
例如：
int  x,  X;
## 变量x 与 X 容易混淆
void foo(int x);   函数foo 与FOO 容易混淆
void FOO(float x);
【规则3-2-2】局部变量不得与全局变量重名，尽管两者的作用域不同而不会发生语法错
误，但会使人误解。
【规则3-2-3】变量的名字应当使用“名词”或者“形容词＋名词”。
例如：
float  value;
float  oldValue;
float  newValue;

【规则3-2-4】全局函数的名字应当使用“动词”或者“动词＋名词”（动宾词组）。各单
词连写，词首字母大写
例如：DrawBox();


【规则3-2-5】用正确的反义词组命名具有互斥意义的变量或相反动作的函数等。
例如：
int minValue;
int maxValue;

int SetValue(…);
int GetValue(…);

3.3 文件命名
【规则3-3-1】项目组应采用相同的风格命名源文件，全小写风格：组成文件名的单词全
采用小写字母，单词间用下划线分隔，避免取一些比较通俗的文件名，如：main.c，public.c
等；首字母大写风格：组成文件名的单词的首字母大写；
3.4 其它命名
【规则3-4-1】数据库对象命名约定采用“项目缩写+数据库类型+版本号”的形式。（注：
这里所指的版本号是指数据库结构的版本号，此版本号可能与系统的版本号不一致）。
例如：股票质押系统英文简称为hssims， 其当前数据库名为simscur10,  历史库名为
simshis10，表名取其英文名。
【规则3-4-2】为保证移植性，嵌入式SQL 语句标识统一用大写的“EXEC SQL”，不允许
用“$”；变量引用统一采用“：”，不允许用“$”。

## 4 语言规范
4.1 常量
【规则4-1-1】需要对外公开的常量放在头文件中，不需要对外公开的常量放在定义文件
的头部。为便于管理，可以把不同模块的常量集中存放在一个公共的头文件中。
【规则4-1-2】如果某一常量与其它常量密切相关，应在定义中包含这种关系，而不应给
出一些孤立的值。
例如：
const  float   RADIUS = 100.0;
const  float   DIAMETER = RADIUS * 2;

4.2 变量
【规则4-2-1】声明变量的同时对变量进行初始化，严禁使用未经初始化的变量。
【规则4-2-2】变量类型的说明应注意机器字长以及应用项目中该变量得取值范围。
4.3 宏
【规则4-3-1】宏体和参数都要加上括号。
例如：
## 正确的宏定义
## 错误的宏定义
#define MAX(a, b)  ((a) > (b) ? (a) : (b))

#define MAX(a, b)  a > b ? a : b


【规则4-3-2】为杜绝可能带来的副作用，尽量避免采用函数宏。
【规则4-3-3】由于const 常量有数据类型，编译器可以对此进行类型安全检查，所以不
## 要用宏来定义常量，而用const。

4.4 表达式
【规则4-4-1】如果代码行中的运算符比较多，用括号确定表达式的操作顺序，避免使用
默认的优先级。
例如：

【规则4-6-3】参数命名要恰当，顺序要合理。
例如编写字符串拷贝函数StringCopy，它有两个参数。如果把参数名字起为str1 和str2，
例如
void StringCopy(char *str1, char *str2);
那么我们很难搞清楚究竟是把str1 拷贝到str2 中，还是刚好倒过来。
可以把参数名字起得更有意义，如叫strSource 和strDestination。这样从名字上就可以
看出应该把strSource 拷贝到strDestination。
还有一个问题，这两个参数那一个该在前那一个该在后？参数的顺序要遵循程序员的习
惯。一般地，应将目的参数放在前面，源参数放在后面。
如果将函数声明为：
void StringCopy(char *strSource, char *strDestination);

【规则4-6-8】return 语句不可返回指向“栈内存”的“指针”，因为该内存在函数体结束
