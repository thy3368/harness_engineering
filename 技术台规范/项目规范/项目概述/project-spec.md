# 项目级概述规范

## 1. 规范概述

本文档定义项目级开发规范，融合 **Spec-Driven Development（规约驱动开发）** 和 **Harness Engineering（测试工程化）** 的优秀实践，为项目开发提供系统化的方法论指导。

### 1.1 规范依据

| 依据来源 | 说明 |
|----------|------|
| Spec-Driven Development (SDD) | 规约优先开发，强调规约作为可信源，人工审查保持意图清晰 |
| Intent-Driven Development | 意图驱动开发，规约需捕获真实设计意图而非仅功能描述 |
| GitHub Spec Kit | 微软 Spec Kit 方法论，结构化规约工作流 |
| Harness Engineering | CI/CD 工程化实践，强调自动化、可靠性、可观测性 |
| INVEST / MoSCoW | 需求拆分框架，用于有意义的特性拆分 |

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **规约优先** | 编写代码前先编写规约，规约是唯一的可信源 |
| **人工可审** | 规约需保持人工可审查，避免因过长而失去审查价值 |
| **最小化规约** | 避免长链式需求，从最小可用规约开始 |
| **有意义拆分** | 将大型特性拆分为独立可交付的价值切片 |
| **持续验证** | 规约与实现持续对齐，自动捕获偏差 |
| **系统思维** | 考虑跨特性交互，防止级联效应 |

## 2. 项目结构规范

### 2.1 多模块 Maven/Gradle 项目

```
project-root/
├── pom.xml / build.gradle              # 父构建文件
├── settings.xml                        # 项目配置
├── 模块名/                              # 核心业务模块
│   ├── src/
│   │   ├── main/java/                  # 生产代码
│   │   ├── main/resources/             # 配置文件
│   │   ├── test/java/                  # 测试代码
│   │   └── test/resources/             # 测试资源
│   └── pom.xml / build.gradle          # 模块构建文件
├── 模块名-api/                          # API 接口模块
├── 模块名-persistence/                 # 数据持久化模块
└── 模块名-config/                      # 配置模块
```

### 2.2 模块分层结构

| 层级 | 目录 | 说明 |
|------|------|------|
| **适配层 (Adapter)** | `adapter/` | 外部系统集成、协议转换 |
| **领域层 (Domain)** | `domain/` | 核心业务逻辑、领域模型 |
| **应用层 (Application)** | `application/` | 用例编排、事务管理 |
| **网关层 (Gateway)** | `gateway/` | 外部网关抽象 |
| **API 层** | `api/` | 接口定义、DTO/VO |

### 2.3 模块依赖原则

```
┌─────────────────────────────────────────────┐
│                 Web 层                      │
│          (Controller / Handler)             │
└─────────────────┬───────────────────────────┘
                  │ 依赖
                  ▼
┌─────────────────────────────────────────────┐
│                API 层                        │
│         (接口定义、DTO/VO)                   │
└─────────────────┬───────────────────────────┘
                  │ 依赖
                  ▼
┌─────────────────────────────────────────────┐
│              Application 层                  │
│             (用例、服务)                      │
└─────────────────┬───────────────────────────┘
                  │ 依赖
                  ▼
┌─────────────────────────────────────────────┐
│               Domain 层                      │
│          (实体、值对象、领域服务)             │
└─────────────────┬───────────────────────────┘
                  │ 依赖
                  ▼
┌─────────────────────────────────────────────┐
│             Adapter 层                      │
│      (外部系统适配、持久化、网关)            │
└─────────────────────────────────────────────┘
```

**依赖规则**：
- 外层依赖内层，内层不依赖外层
- Domain 层为核心，不依赖任何外部层
- Adapter 层可被依赖但不产生依赖
- 禁止跨层依赖（如 Controller 直接依赖 Persistence）

## 3. 规约驱动开发规范

### 3.1 规约编写原则

| 规则 | 说明 |
|------|------|
| **人类可审查** | 规约必须可由人工完整审查，单个规约不超过 500 行 |
| **意图优先** | 捕获设计意图而非仅功能描述，说明"为什么" |
| **最小化开始** | 从最小可用规约开始，避免预定义未来需求 |
| **结构化拆分** | 使用 INVEST 原则拆分用户故事 |
| **版本化规约** | 规约采用语义化版本管理 |

### 3.2 规约文档结构

```markdown
# [特性名称]

## 1. 概述
- 特性描述
- 业务价值
- 适用范围

## 2. 功能需求
### 2.1 核心功能
- 功能点描述
- 输入/输出定义
- 业务流程

### 2.2 边界条件
- 异常场景处理
- 性能约束

## 3. 非功能需求
- 性能要求
- 安全要求
- 可用性要求

## 4. 交互设计
- 接口定义
- 数据格式
- 错误码规范

## 5. 验收标准
- 功能验收条件
- 测试用例
```

### 3.3 规约与代码对齐

| 规则 | 说明 |
|------|------|
| **唯一可信源** | 规约是设计的唯一可信来源 |
| **持续验证** | 代码变更需同步更新规约 |
| **差异追踪** | 实现与规约的差异需明确记录 |
| **自动化检查** | 关键约束通过自动化测试验证 |

### 3.4 规约编写检查清单

- [ ] 规约是否清晰描述了业务意图？
- [ ] 规约是否可以人工完整审查？
- [ ] 规约是否拆分为有意义的独立切片？
- [ ] 规约是否包含明确的验收标准？
- [ ] 规约是否考虑了跨特性交互？
- [ ] 规约是否避免了"and"链式长句？

## 4. 测试工程化规范

### 4.1 测试分层

| 测试类型 | 覆盖率目标 | 执行频率 | 责任人 |
|----------|------------|----------|--------|
| **单元测试** | 核心业务 > 80% | 每次提交 | 开发人员 |
| **集成测试** | 关键路径 100% | 每次构建 | 开发人员 |
| **端到端测试** | 核心流程 | 每日构建 | QA/开发 |
| **性能测试** | 关键路径 | 每周/发布 | 性能团队 |

### 4.2 测试金字塔

```
                    ┌─────────────┐
                    │   E2E 测试   │
                    │    (5-10%)   │
                    └──────┬──────┘
               ┌────────────┼────────────┐
               │      集成测试         │
               │       (20-30%)       │
               └────────────┬────────────┘
        ┌───────────────────┼───────────────────┐
        │              单元测试                  │
        │             (60-70%)                  │
        └───────────────────────────────────────┘
```

### 4.3 测试命名规范

| 模式 | 格式 | 示例 |
|------|------|------|
| **单元测试** | `方法名_场景_预期结果` | `transfer_invalidAmount_throwsException` |
| **集成测试** | `功能_场景_验证点` | `transfer_api_concurrent_success` |
| **E2E 测试** | `场景_验证结果` | `user_registration_complete_flow` |

### 4.4 测试工程化检查清单

- [ ] 是否有清晰的测试分层策略？
- [ ] 测试是否遵循 AAA 模式（Arrange-Act-Assert）？
- [ ] 测试是否具有独立的执行环境？
- [ ] 测试是否避免了对外部系统的强依赖？
- [ ] 测试数据是否可重复、可清理？
- [ ] 测试覆盖率是否达到目标？

## 5. CI/CD 工程化规范

### 5.1 流水线阶段

| 阶段 | 任务 | 工具 |
|------|------|------|
| **代码检出** | 拉取代码 | Git |
| **依赖解析** | 下载依赖 | Maven/Gradle |
| **代码编译** | 编译构建 | Compiler |
| **静态分析** | 代码检查 | SonarQube |
| **单元测试** | 执行单元测试 | JUnit/TestNG |
| **构建镜像** | Docker 镜像构建 | Docker |
| **部署测试环境** | 部署到测试环境 | K8s/Helm |
| **集成测试** | 执行集成测试 | Postman/JMeter |
| **部署预发布** | 部署到预发布环境 | ArgoCD/Flux |
| **部署生产** | 部署到生产环境 | Harness |

### 5.2 部署策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **蓝绿部署** | 两套环境切换 | 无状态服务 |
| **金丝雀发布** | 逐步流量切换 | 有状态服务 |
| **滚动更新** | 逐实例更新 | K8s 默认 |
| **特性开关** | 运行时切换特性 | 特性灰度 |

### 5.3 回滚策略

| 规则 | 说明 |
|------|------|
| **自动回滚** | 关键指标异常时自动回滚 |
| **手动回滚** | 提供一键回滚能力 |
| **回滚验证** | 回滚后自动验证服务可用性 |
| **回滚记录** | 记录回滚原因和影响范围 |

## 6. 可观测性规范

### 6.1 日志规范

| 级别 | 使用场景 |
|------|----------|
| **ERROR** | 程序异常，需人工介入 |
| **WARN** | 潜在问题，可恢复异常 |
| **INFO** | 关键业务流程节点 |
| **DEBUG** | 开发调试信息 |

**日志格式规范**：
```json
{
  "timestamp": "2026-03-31T10:00:00.000Z",
  "level": "INFO",
  "traceId": "abc123",
  "spanId": "def456",
  "service": "order-service",
  "method": "createOrder",
  "message": "Order created successfully",
  "context": {
    "orderId": "ORD-20260331001",
    "userId": "USR-001"
  }
}
```

### 6.2 指标规范

| 指标类型 | 示例 | 告警阈值 |
|----------|------|----------|
| **延迟** | p99 < 200ms | > 500ms |
| **错误率** | < 0.1% | > 1% |
| **吞吐量** | > 1000 QPS | < 500 QPS |
| **资源使用** | CPU < 70% | > 85% |

### 6.3 链路追踪

- 全链路追踪覆盖所有关键路径
- Trace ID 跨服务传递
- Span 包含足够上下文用于问题定位

## 7. 安全规范

### 7.1 安全开发原则

| 原则 | 说明 |
|------|------|
| **纵深防御** | 多层安全防护 |
| **最小权限** | 权限按需分配 |
| **安全默认** | 默认配置安全 |
| **审计追溯** | 关键操作记录日志 |

### 7.2 敏感数据处理

| 数据类型 | 处理要求 |
|----------|----------|
| **密码** | 加密存储，使用 BCrypt |
| **密钥** | 密钥管理服务存储 |
| **个人隐私** | 脱敏处理，符合 GDPR |
| **传输数据** | TLS 1.3 加密 |

### 7.3 安全检查清单

- [ ] 输入验证是否覆盖所有入口点？
- [ ] 敏感数据是否加密存储？
- [ ] 认证授权是否正确实现？
- [ ] SQL/命令注入是否防护？
- [ ] 日志是否脱敏处理？

## 9. 版本与发布规范

### 9.1 版本命名

采用语义化版本 `MAJOR.MINOR.PATCH`：
- **MAJOR**: 不兼容的 API 变更
- **MINOR**: 向后兼容的功能新增
- **PATCH**: 向后兼容的问题修复

### 9.2 发布检查清单

- [ ] 所有测试通过
- [ ] 静态分析无高危问题
- [ ] 安全扫描无漏洞
- [ ] 性能测试达标
- [ ] 文档已更新
- [ ] 变更日志已记录

| 类型 | 内容 | 位置 |
|------|------|------|
