# 数据库规范

## 概述

本规范基于 [恒生数据库编码规范](./恒生数据库编码规范.md)，用于指导项目数据库设计与开发。

## 命名规范

### 对象命名

| 规则 | 说明 |
|------|------|
| **强制** | 禁止使用数据库关键字和保留字进行命名 |
| **推荐** | 对象的命名由三部分组成：对象前缀+模块名+对象标识，使用下划线"_"进行分隔，长度不超过30个字符 |
| **推荐** | 命名只能使用英文字母、数字和下划线，全部使用小写字符 |
| **推荐** | 命名尽量采用有意义且简短的英文单词，最多使用四个单词 |
| **推荐** | 字段命名时，含义要明确，避免使用单个 `id`，应使用 `user_id` 等 |
| **推荐** | 表达"是否"概念的字段，建议使用 `is_` 作为前缀，如 `is_deleted` |

详见 [恒生数据库编码规范 - 对象命名](./恒生数据库编码规范.md#对象命名)

### 变量和参数命名

| 规则 | 说明 |
|------|------|
| **强制** | 除循环变量外，禁止使用单个字符（如 `i`、`j`、`k`）命名变量 |
| **推荐** | 本地变量以 `v_` 为前缀，参数以 `p_` 为前缀 |

## 设计规范

### 通用部分

| 规则 | 说明 |
|------|------|
| **推荐** | 字段应选择合适的数据类型及长度，尽量使用 `varchar` 代替 `char` |
| **推荐** | 建表时字段按顺序排列：主键字段 > 常用字段 > 短小字段 > 非空字段 > 其他字段 |
| **推荐** | 遵循三范式设计原则，个别情况下可考虑反范式设计 |
| **推荐** | 表上的索引建议不超过5个，每个索引的字段数建议不超过5个 |
| **强制** | 禁止使用外键，外键约束应该在应用层处理 |
| **推荐** | 主键统一使用无符号整型 |

### MySQL 特有

| 规则 | 说明 |
|------|------|
| **推荐** | 使用 InnoDB 存储引擎 |
| **推荐** | 字符集统一使用 UTF8MB4 |
| **强制** | 所有表都必须有主键 |
| **强制** | 禁止使用 `enum`/`set`/`bool` 类型 |
| **推荐** | 尽量不使用 `text`/`blobs` 等大字段类型 |
| **强制** | 禁止使用浮点类型 `FLOAT` 和 `DOUBLE` |
| **推荐** | 对较长的字符型字段建立索引时，应使用前缀索引 |
| **推荐** | 字段建议定义为 `not null`，并且指定默认值 |

### Oracle 特有

| 规则 | 说明 |
|------|------|
| **推荐** | 表和索引的数据分不同表空间存储 |
| **推荐** | 优先使用业务主键，在业务主键超过3个字段时考虑使用逻辑主键 |
| **推荐** | 不建议在分区表上创建全局索引，原则上一律使用 local 索引 |
| **推荐** | 字段类型使用字符型（兼容 MySQL 和 Oracle） |

## 语法规范

### 通用部分

| 规则 | 说明 |
|------|------|
| **强制** | 使用 SQL99 语法标准，在多表关联时用 `join` 关键字 |
| **强制** | 禁止 `select *`，必须将字段名一一列出 |
| **强制** | `insert` 语句中必须列出要插入的字段名 |
| **强制** | 当 SQL 涉及到多个表时，必须为每个字段指定表名前缀 |
| **强制** | 不需要去除重复数据时使用 `union all`，而不是 `union` |
| **推荐** | 尽量避免使用 `or` 操作符 |
| **强制** | `update` 和 `delete` 语句必须有 `where` 子句 |
| **推荐** | 使用 `like` 进行模糊查询时，`%` 不要放在首位 |
| **推荐** | 原则上禁止使用触发器 |
| **强制** | 外连接一律用 `left join`，禁止使用 `right join` |
| **推荐** | 尽量使用内连接，减少使用外连接 |

### MySQL 特有

| 规则 | 说明 |
|------|------|
| **推荐** | 避免三个表以上的关联查询，避免大表间的 join |
| **推荐** | 联表查询时，连接列的数据类型必须一致 |
| **推荐** | 不建议使用存储过程、函数 |

### Oracle 特有

| 规则 | 说明 |
|------|------|
| **推荐** | 尽量使用静态 SQL，少用动态 SQL，使用动态 SQL 时要注意绑定变量 |
| **强制** | 对于动态 SQL，新研发产品应该强制使用绑定变量 |
| **推荐** | 原则上避免使用游标变量，必须使用时要及时关闭游标 |
| **推荐** | 不要随意使用 `commit`，应保证事务完整性 |
| **强制** | 禁止使用 `goto` 语句来控制流程 |

## 格式规范

### 缩进

| 规则 | 说明 |
|------|------|
| **强制** | 代码块采用缩进风格书写，缩进格数为2个空格，禁止使用 tab 键 |

### 大小写

| 规则 | 说明 |
|------|------|
| **推荐** | 除字符串外，统一使用小写字符书写 |

### 换行

| 规则 | 说明 |
|------|------|
| **强制** | 一行只写一条语句 |
| **推荐** | 同一条语句中关键字右对齐 |
| **推荐** | 相对独立的程序块之间需要用空行隔开 |
| **推荐** | 对超过120字符的语句要分行书写 |
| **强制** | `begin`、`end` 等关键字独立成行 |

### 空格

| 规则 | 说明 |
|------|------|
| **推荐** | 操作符前后应以空格分隔 |
| **推荐** | 分隔符之后应以空格分隔 |

### 注释

| 规则 | 说明 |
|------|------|
| **推荐** | 脚本文件、函数、过程头部和代码修改处应进行注释 |
| **强制** | 注释应紧挨其描述的代码 |
| **强制** | 注释与所描述的代码进行同样的缩进 |

## 脚本规范

### 脚本规范总则

| 规则 | 说明 |
|------|------|
| **推荐** | 脚本必须支持重复执行不报错 |
| **推荐** | 脚本名需要加上执行用户作为前缀 |
| **强制** | 在修改代码的上一行书写注释，包括：修改日期，修改单编号/QC号：修改说明 |
| **强制** | 对于业务依赖的修改，应在代码逻辑上保证其依赖性 |

### MySQL 特有

| 规则 | 说明 |
|------|------|
| **推荐** | 脚本中需要判断执行数据库是否正确 |
| **推荐** | 代码中避免使用反引号 "`" |

### Oracle 特有

| 规则 | 说明 |
|------|------|
| **强制** | PLSQL 匿名块脚本后需要增加 `/` |
| **推荐** | 文件头加上 `set define off;`，文件尾加上 `set define on;` |
| **推荐** | 脚本中需要判断执行用户是否正确 |

### 数据库对象

| 规则 | 说明 |
|------|------|
| **强制** | 单个表的表结构修改代码写在一个文件中 |
| **强制** | 对数据库对象的创建、修改、删除和数据的新增、修改前，应先进行存在性判断 |
| **推荐** | 对表进行增加字段时，应同时加上字段备注 |

### 表数据

| 规则 | 说明 |
|------|------|
| **强制** | 单个表的表数据修改代码写在一个文件中 |
| **强制** | 在书写 `update` 和 `delete` 语句时，必须带有 `where` 条件 |

## 版本移植

| 规则 | 说明 |
|------|------|
| **推荐** | 对于单行注释应使用 `-- ` 方式 |
| **强制** | 对于字符串应该使用单引号括起来 |
| **推荐** | 尽量避免使用 `full join` |
| **推荐** | 避免在 `group by` 和 `having` 子句中使用列的别名 |
| **推荐** | `from` 子句中的子查询要定义别名 |
