# MC3.0 DFX 规格说明书

## 概述

> **测试场景**: 1000个主题（单分区），1000个生产者 100个消费者（一对一，不交叉订阅）1KB报文  
> **部署结构**: 1主1备  
> **配置要求**: 使用发布包里面的标准配置

---

## 1. 性能指标

### 1.1 时延指标

| 指标 | 标准值 | 说明 |
|------|--------|------|
| 穿透时延 | `<1ms` | 标准场景下的单笔延时 |

### 1.2 吞吐指标

| 指标 | 标准值 | 说明 |
|------|--------|------|
| 消息推送能力 | `5W tps` | 推送总数，单主题≥3W |
| 客户端发布能力 | 单生产者≥3W tps | |
| 客户端消费能力 | 单消费者≥3W tps | |

### 1.3 容量指标

| 指标 | 标准值 | 说明 |
|------|--------|------|
| 主题数量 | ≤4096 | |
| 生产者数量 | 1000 | |
| 消费者数量 | 10W | |
| 订阅项数量 | 20W | 单节点 |
| 单个生产者/消费者连接数 | ≤1024 | |
| 单个消费者会话数 | ≤1024 | |
| 消息大小上限 | 1GB | 建议1KB最佳 |
| 消息容量 | 不限 | 取决于磁盘大小 |

### 1.4 主题配置指标

| 指标 | 标准值 | 说明 |
|------|--------|------|
| 单个主题分区数 | ≤1024 | 至少1个 |
| 单个主题分区副本数 | ≤16 | |
| 主题名长度 | ≤256字节 | 不包含结尾`\0` |
| 过滤条件数 | ≤16 | |
| 过滤条件名长度 | ≤63字节 | 不包含结尾`\0` |
| 单分区生产者持久化文件数 | ≤256 | |
| 服务集群名长度 | ≤64字节 | 包含结尾`\0` |
| 服务集群成员个数 | ≤64 | 实际依赖RCM集群成员个数 |

### 1.5 横向扩展

#### 发布性能

- 引入主题分区提高发布性能横向扩展能力

**使用约束**:
1. 需提前预估单分区发布性能是否满足成交回报性能要求
2. 如需消息全局顺序，需配置主题属性为全局有序（此时单分区性能为上限）
3. 设置多分区后，业务发布者需设置分区路由规则（如 `业务账号hash % 分区数`）
4. 盘中修改主题分区数会导致无法保证分区有序

#### 订阅推送性能

- 引入推送线程和分区副本提升订阅推送横向扩展能力

**使用约束**:
1. 单个推送线程性能决定单个消费者订阅某主题分区的推送性能
2. 副本数（最多16个）越多越影响发布性能，需权衡
3. 订阅项总数巨量时（50W+），需业务配合设置主题分区级订阅（未实现）

#### 分区重分配

- 按 `消息索引 > 同机房 > 优先级 > 负载均衡` 维度进行分区重分配
- 启动时进行一次分区重分配（可配置关闭）

### 1.6 超过性能上限风险

| 场景 | 风险描述 | 建议 |
|------|----------|------|
| 订阅项数量超限 | 订阅爆慢，极端情况下客户端订阅速度超过服务端处理速度，导致恶性循环、消息积压 | 横向扩展MC集群节点 |
| 系统吞吐超限 | 服务端消息积压，极端情况下积压数超过消息池消息总数（100W），存在丢失推送消息风险 | 监控推送队列情况，积压>1000时横向扩展 |

## 2. 可靠性指标

### 2.1 高可用要求

| 指标 | 标准值 | 说明 |
|------|--------|------|
| RTO | 5s | 恢复时间目标 |
| RPO | 0 | 恢复点目标（未实现） |

### 2.2 网络基础要求

1. **网络质量**: 延时≤30ms，丢包率<1%
2. **断连恢复**: 客户端间隔10s重连，20s内重连成功；需心跳识别的场景90s内完成
3. **数据目录损坏**: 启动时会导致MC3.0启动失败，需删除数据重启或重装
4. **资源水位恢复**: 内存/磁盘/CPU超过水位后，30s内恢复正常
5. **节点重新加入**: 需数据同步完成后才能提供服务
6. **启动就绪**: 节点同步完成变成ServerReady后，客户端可正常收发
7. **资源利用率**: CPU/内存/IO/磁盘 <90%，超限会导致启动时间过长或运行时异常

### 2.3 集群模式

#### 单机房多节点强一致（未实现）

- 网络中断/网卡故障：节点设为孤岛，客户端断开切换到其他节点
- 节点宕机：可用节点>1时客户端自动重连，否则集群不可用
- 异步刷盘模式：掉电会丢失消息

#### 单机房多节点最终一致

- 网络中断/网卡故障：节点设为孤岛，客户端断开切换，网络恢复后直接恢复正常
- 节点宕机：客户端自动重连，有节点运行则整体不受影响
- 异步刷盘模式：掉电会丢失消息

#### 多网卡

- 支持配置2个以上网段，生产者和消费者可在不同网段进行消息收发

#### 仲裁

双机房集中仲裁方案，主备机房切换时优先选择同机房节点：
1. 主题分区主节点优先使用主机房MC节点
2. 客户端订阅时优先订阅同机房MC节点
3. 集中仲裁期间切主优先切换到同机房其他节点

### 2.4 接口容错要求

| 接口 | 容错行为 |
|------|----------|
| LDP消息容错 | 消息格式错误，断开连接 |
| 发布接口容错 | 1. 分区备不接受发布，返回报错<br>2. 无发布权限，返回报错（未实现）<br>3. 发布必填字段校验（主题） |
| 订阅接口容错 | 1. 未建立会话，返回报错<br>2. MC版本号不一致，返回报错<br>3. 无订阅权限，返回报错（未实现）<br>4. 订阅必填字段校验（主题） |

### 2.5 数据保护

| 指标 | 说明 |
|------|------|
| 消息正确率100% | 开机补缺 + 集群间补缺 |
| 消息乱序率 | 0 |
| 消息丢失率 | 0 |

**开机补缺**: 启动时校验本地持久化数据，尝试从其他节点同步最新数据，发现破坏则尝试修复

**集群间补缺**: 主题分区发布时，如RCM同步数据不连续，尝试从其他节点拉取数据更新

### 2.6 系统稳定性

- 目前无法完全达到7*24
- 服务端支持过期消息清理可达到7*24
- 客户端暂不支持消息序号文件清理，无法达到7*24

## 4. 资源异常

### 4.1 CPU使用率

- 超过90%会导致启动时间过长或失败
- 运行时会出现连接限流、响应超时
- 停止时可能因停止过慢被OS kill

### 4.2 内存占用

- 主要是消息池：每个消息占用20KB，1W消息200MB
- 服务端还有mmap文件带来的内存占用
- 内存不足可能被OS kill

### 4.3 磁盘占用

| 场景 | 影响 |
|------|------|
| 生产者磁盘不足 | 发布缓存文件创建失败，发布接口返回失败 |
| MC节点磁盘不足 | 消息/索引文件创建失败，返回发布失败 |
| 消费者磁盘不足 | 消费偏移量文件保存失败，可能重复消费 |

### 4.4 磁盘I/O

文件级主题会写磁盘文件：
1. **生产者发布**: 同步写磁盘，IO下降则生产者队列积压
2. **服务端收消息**: 异步刷盘写消息文件，IO下降则发布队列积压、重启有丢消息风险
3. **服务端集群复制**: 异步刷盘写消息文件，IO下降则复制队列积压、重启有丢消息风险
4. **消费者消费成功**: 同步更新序号文件，IO下降则消费队列积压

### 4.5 网络I/O

| 场景 | 影响 |
|------|------|
| 生产者→服务端 | tcp |
| 服务端集群内指令分发 | 组播（rcm）-- 故障影响仲裁和主题分区切换 |
| 服务端集群内消息复制 | 组播（rcm）-- 故障影响数据一致性 |
| 服务端开机补缺/集群应答 | 单播（rcm）-- 故障影响数据一致性 |
| 服务端→消费者 | tcp -- 断开20秒以上会话断开，所有订阅项取消 |

### 4.6 文件句柄

- 消息文件、索引文件、socket句柄
- 不足时：服务端拒绝新连接、文件和消息创建失败

### 4.7 线程数量（默认）

| 角色 | 线程 |
|------|------|
| 服务端 | 主题服务1 + 监控1 + 仲裁1 + 发布4 + 补缺5 + 订阅4 + 推送4 + 会话心跳1 + 会话重传1 + 会话4 + 订阅补缺1 + rcm上下文4×3 + remoter2 + 框架主线程1 |
| 单个生产者 | 定时1 + 接收1 + remoter1 + 框架主线程1 |
| 单个消费者 | 会话工作1 + 回调1 + 会话监控1 + remoter1 + 框架主线程1 |

### 4.8 连接数与端口

- **连接数**: 最多 消费者数 + 生产者数 + 集群节点
- **端口数**: 对外发布/订阅1 + 管理1 + rcm组播/单播组3×2

## 8. 可服务性

### 8.1 环境要求

| 规格 | 配置 |
|------|------|
| 标准配置 | 16核，3.0GHz主频，128G内存，500G SSD，万兆网卡 |
| 最高配置 | - |
| 最低配置 | - |

### 8.2 部署方式

| 部署模式 | 说明 |
|----------|------|
| 单点部署 | mcsdk直连，性能最佳，延时<10微秒 |
| 集群部署 | 通过MC3.0服务端，无强一致单分区延时10~20微秒，强一致多分区延时<100微秒 |
| 双机房多活 | 需要ZK |
| 跨地域部署 | 网络代理+ZK（未实现） |

**组件部署**:
- mcproxy、mc3_publish_to_mc2、mc3stomp: 同mcsdk部署方式
- minimc: 依赖front_mc，部署在一起，可独立于MC服务端

### 8.3 版本升级/迁移

- 版本向下兼容
- 用于兼容替换异构系统：需按MC3.0规则根据现有业务要求重新配置
- 版本升级迭代：根据修改点具体评估

## 9. 备份恢复

| 类型 | 备份位置 | 恢复说明 |
|------|----------|----------|
| 生产者 | 工作路径/mc30/集群名/McVersion/实例名 | 恢复该目录即可，MC被清理过则失败（版本号对不上） |
| 消费者 | 工作路径/集群名/McVersion/global_map_集群名.dat | 恢复该文件即可，MC被清理过则失败 |
| 服务端 | 工作目录/mcstore/default | 恢复该目录即可，MC被清理过则失败 |

## 10. 应急处置（未实现）

1. **磁盘空间不够/节点无法启动**: 支持将主题等级设置为内存级
2. **MC节点不可用**: 支持消费者直接对生产者发起端对端消费
3. **消费者消息缺失**: 支持将特定消费者设置为强制消费历史消息
4. **服务端数据缺失**: 支持将特定生产者设置为再次发起消息发布
5. **节点无法启动**: 支持迁移数据文件至新节点
6. **数据一致性监控**: 5分钟内数据未变化则告警
