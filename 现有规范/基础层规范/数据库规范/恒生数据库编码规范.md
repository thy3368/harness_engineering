命名规范
对象命名
1. **推荐** 对象的命名由三部分组成：对象前缀+模块名+对象标识，使用下划线“_”进行分隔，长度不超过30个字符。例如：v_crm_custinfo。下表列出了对象前缀命名原则：
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
2. **推荐** 命名只能使用英文字母、数字和下划线，全部使用小写字符。
禁止以数字开头，名称前后不能加引号。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
3. **推荐** 命名尽量采用有意义且简短的英文单词，最多使用四个单词。
多个单词组成的，中间以下划线分隔。
在不产生歧义的前提下可以使用单词简写，如：info ,cfg等。
建议检查阶段：设计评审
检查方式：人工检查
4. **推荐** 字段命名时，含义要明确。
比如user表的用户编号字段，不能仅仅命名为id，应命名为user_id。好处是在进行多表关联时更容易理解，比如user.user_id = role.user_id。
表达“是否”概念的字段，建议使用 is_作为前缀，如“是否删除”应命名为is_deleted。
建议检查阶段：设计评审
检查方式：人工检查
5. **推荐** 字段命名时避免数字在下划线之后。
理由：避免Jres工具驼峰转下划线出问题。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
6. **强制** 禁止使用数据库关键字和保留字进行命名。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
7. **推荐** 不使用复数形式的名词，但只有复数形式的名词除外。
必须使用复数形式的词，例如：news（新闻）、times（次数）。
字段包含多个以逗号分隔的值时，应加后缀_list。比如客户权限，值为’a,b,c’，应命名为client_privilege_list。
建议检查阶段：设计评审
检查方式：人工检查
变量和参数命名
1. **推荐** 变量和参数的命名格式如下：变量前缀+变量标识。其中变量标识的命名方式同字段命名。本地变量以v_为前缀，参数以p_为前缀。例如v_begin_date。
建议检查阶段：代码提交
检查方式：扫描工具
2. **强制** 除循环变量外，禁止使用单个字符（如i、j、k）命名变量。
建议检查阶段：代码提交
检查方式：人工检查

空格
1. **推荐** 操作符前后应以空格分隔。示例：

应写成
set v_error_no = 0
2. **推荐** 分隔符之后应以空格分隔。示例：
select 1,2,3
应写成
select 1, 2, 3
建议检查阶段：代码提交
检查方式：扫描工具

建议检查阶段：代码提交
检查方式：人工检查
2. **强制** 注释应紧挨其描述的代码，在其描述代码的上方或者右方。
建议检查阶段：代码提交
检查方式：人工检查
3. **推荐** 注释如放在代码段的上方则应与其上面的代码段用空行隔开。
建议检查阶段：代码提交
检查方式：人工检查
4. **强制** 注释与所描述的代码进行同样的缩进。
建议检查阶段：代码提交
检查方式：扫描工具
5. **推荐** 通过对函数、过程、变量等进行合理命名，使代码成为自注释的。
建议检查阶段：代码提交
检查方式：人工检查

设计规范
通用部分
1. **推荐** 字段应选择合适的数据类型及长度。
尽量不要用char类型，用varchar类型代替。仅当存储的字符是相同、固定的长度且有非数字时使用char类型。
建议检查阶段：设计评审
检查方式：人工检查
2. **推荐** 建表时字段应按下面的顺序从前往后排列：
主键字段 > 常用字段 > 短小字段 > 非空字段 > 其他字段。
建议检查阶段：设计评审
检查方式：人工检查
3. **推荐** 遵循三范式设计原则，个别情况下可考虑反范式设计。
建议检查阶段：设计评审
检查方式：人工检查
4. **推荐** 表上的索引建议不超过5个，每个索引的字段数建议不超过5个。
索引应该由DBA或技术经理统一管理维护，禁止开发和实施人员私自增加、改动。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
5. **强制** 禁止使用外键，外键约束应该在应用层处理。建议在ER图里保留外键。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
6. **推荐** 主键统一使用无符号整型，mysql设置该字段为auto_increment自增，oracle 12c 使用GENERATED ALWAYS AS IDENTITY新特性定义该字段。
理由：兼容oracle和mysql数据库，只维护一份表结构。业务sql也便于兼容。
建议检查阶段：设计评审
检查方式：人工检查
7. **推荐** 应用程序中原则上禁止对主键字段进行更新，如需更新，需要显性通过产品架构师审批。
建议检查阶段：设计评审
检查方式：人工检查

Oracle特有
1. **推荐** 表和索引的数据分不同表空间存储。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
2. **推荐** 原则上每个表都应该有主键。
优先使用业务主键，在业务主键超过3个字段时考虑使用逻辑主键，把业务主键建成唯一约束。
建议检查阶段：设计评审
检查方式：人工检查
3. **推荐** 不建议在分区表上创建全局索引，原则上一律使用local索引。
建议检查阶段：设计评审、代码提交
检查方式：扫描工具
4. **推荐** 字段类型使用字符型。
理由：产品需要同时兼任多种数据库，oracle字段类型默认为字节型，mysql为字符型。为保证设计字段长度不会导致mysql过长，或者oracle长度不足，因此统一为字符型。如字符集都为utf8，mysql设计为varchar(2)可以存储两个汉字，而oracle设计为varchar2(2)不足以存储一个汉字。
建议检查阶段：设计评审
检查方式：人工检查

脚本规范
脚本规范总则
通用部分
1. **推荐** 脚本必须支持重复执行不报错。
理由：不影响集成和测试等多次升级。
建议检查阶段：代码集成
检查方式：扫描工具
2. **推荐** 脚本名需要加上执行用户作为前缀，以“.sql”文件格式存储。
理由：便于及时发现脚本提交路径和执行用户不正确。
建议检查阶段：代码提交
检查方式：扫描工具
3. **推荐** 数据库中加入特定系统参数表标识数据库环境（如生产、测试），在执行dml、ddl脚本时先判断此参数是否吻合。
理由：防止诸如清库等脚本在错误的环境执行。
建议检查阶段：代码提交
检查方式：人工检查
4. **强制** 在修改代码的上一行书写注释，包括：修改日期，修改单编号/QC号：修改说明。
-- 20210506 Mxxxxxxxx：新增字段xxxxxxxxx
-- 20210507 Mxxxxxxxx：新增索引xxxxxxxxx
建议检查阶段：代码提交
检查方式：人工检查
5. **强制** 对于业务依赖的修改，应在代码逻辑上保证其依赖性。
理由：防止上一步执行失败后，继续执行下一步操作。
建议检查阶段：代码提交
检查方式：人工检查
6. **推荐** 在脚本前部应打印脚本名称或标记。
理由：在升级报错时能快速定位。
建议检查阶段：代码提交
检查方式：人工检查
7. **推荐** 在升级前对重要数据进行统一备份，其他数据可在脚本中自行备份。
理由：便于升级异常后的数据恢复。
建议检查阶段：代码提交
检查方式：人工检查
8. **推荐** 升级脚本集成时，需要将脚本执行日志落地到本地文件。
理由：便于后期对升级进行回溯。
建议检查阶段：代码集成
检查方式：人工检查
9. **推荐** 对执行耗时较长的脚本，在执行前进行提示。
理由：提高升级友好性。
