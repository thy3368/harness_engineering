# 通用编码规范

## 概述

本规范参考 Google、Microsoft、Meta、Netflix 等大厂工程实践，以及 lifedever/claude-rules、CLAUDE.md 等开源项目，为 AI 编码助手提供通用约束。

> **核心原则**：技术事实和数据优先于个人偏好；代码健康度持续提升而非追求完美。

详见 [Google Eng Practices](https://google.github.io/eng-practices/)

## 量化指标

| 指标 | 限制 | 来源 |
|------|------|------|
| 单个函数 | ≤30 行 | lifedever/claude-rules |
| 单个文件 | ≤300 行 | lifedever/claude-rules |
| 函数参数 | ≤4 个 | lifedever/claude-rules |
| 嵌套层次 | ≤3 层 | lifedever/claude-rules |
| 循环/分支 | ≤3 层 | lifedever/claude-rules |

## 代码组织原则

### 模块化设计

| 规则 | 说明 |
|------|------|
| **强制** | 遵循单一职责原则（SRP） |
| **强制** | 明确分离关注点，保持代码职责清晰 |
| **强制** | 使用函数和类将问题分解为自包含单元 |
| **强制** | 确保组件模块化和可重用性 |
| **强制** | 禁止出现两条等价的支路 |

### 架构原则

| 规则 | 说明 |
|------|------|
| **强制** | 依赖倒置原则：面向接口编程 |
| **强制** | 遵循 DRY 原则，避免重复代码 |
| **强制** | 遵循 YAGNI 原则，不过度设计 |
| **强制** | 遵循 KISS 原则，选择简单直接的方案 |
| **推荐** | 优先使用组合而非继承 |

### 遗留代码态度

| 规则 | 说明 |
|------|------|
| **核心** | 不模仿遗留代码中的坏习惯，应主动重构 |
| **强制** | 新代码必须使用现代 API 和最佳实践 |
| **强制** | 发现不符合规范时，应主动重构而非盲目遵循 |
| **推荐** | 如有时间，逐步改善遗留代码 |

## 编码风格

### 格式规范

| 规则 | 说明 |
|------|------|
| **强制** | 使用 4 空格缩进（禁止 TAB） |
| **强制** | 代码行保持 80-100 字符 |
| **强制** | 使用空行分隔逻辑代码块 |
| **强制** | 尾部保留一行空行 |
| **强制** | 遵循项目 style guide（Google Style 等） |

### 命名规范

| 类型 | 规则 | 示例 |
|------|------|------|
| 类名 | 名词/名词短语 | `UserService`, `OrderRepository` |
| 接口名 | 形容词+名词 | `Readable`, `Serializable` |
| 方法名 | 动词/动词短语 | `getUserById`, `calculateTotal` |
| 常量 | 全大写+下划线 | `MAX_RETRY_COUNT` |
| 布尔变量 | is/has/can 前缀 | `isValid`, `hasPermission` |
| 变量 | 小驼峰，见名知意 | `userList`, `totalAmount` |

| 规则 | 说明 |
|------|------|
| **强制** | 使用有意义且描述性强的名称 |
| **强制** | 避免单字母缩写（i/j 循环变量除外） |
| **强制** | 同一项目相同概念使用一致命名 |
| **强制** | 不使用硬编码的数字或字符串，使用常量 |

### 注释规范

| 规则 | 说明 |
|------|------|
| **强制** | 公共 API 必须有文档注释 |
| **强制** | 多行注释使用 `/* */`，单行使用 `//` |
| **强制** | 修改代码必须添加注释，说明时间、原因、修改人 |
| **推荐** | 保持注释与代码一致，边写代码边加注释 |
| **推荐** | 解释非显而易见的逻辑，而非重复代码 |
| **禁止** | 禁止对不言自明的代码过度注释 |

## 函数设计

| 规则 | 说明 |
|------|------|
| **强制** | 函数只做一件事 |
| **强制** | 函数不超过 30 行 |
| **强制** | 参数不超过 4 个 |
| **强制** | 函数入口点唯一（单一出口） |
| **强制** | 使用明确的返回类型，避免 void 返回 |
| **推荐** | 函数名清晰表达其功能 |
| **推荐** | 避免布尔参数，使用两个函数替代 |

## 错误处理

| 规则 | 说明 |
|------|------|
| **强制** | 异常信息清晰描述错误原因和位置 |
| **强制** | 禁止空 catch 块 |
| **强制** | 资源使用 try-with-resources 或 defer 释放 |
| **强制** | 不捕获 Exception 后直接忽略 |
| **推荐** | 自定义异常继承适当的基类 |
| **推荐** | 区分可恢复和不可恢复异常 |
| **推荐** | 使用 Result 类型替代异常（Go/Rust） |

## 安全规范

| 规则 | 说明 |
|------|------|
| **强制** | 防止 SQL 注入，使用参数化查询 |
| **强制** | 防止 XSS，对输出进行转义 |
| **强制** | 防止 CSRF，使用 Token 验证 |
| **强制** | 禁止硬编码敏感信息（密钥、密码） |
| **强制** | 使用安全的随机数生成器 |
| **强制** | 遵循最小权限原则 |
| **推荐** | 使用 HTTPS 而非 HTTP |

## 测试规范

### 测试覆盖

| 规则 | 说明 |
|------|------|
| **强制** | 覆盖正常情况、边界情况、异常情况 |
| **强制** | 单元测试必须通过才能提交 |
| **强制** | 测试必须具有确定性（deterministic） |
| **推荐** | 遵循测试金字塔原则 |
| **推荐** | 测试保持独立性和可重复性 |

### 测试命名

| 规则 | 说明 |
|------|------|
| **强制** | 测试名描述预期行为 |
| **强制** | 使用 Given-When-Then 模式 |
| **强制** | 每个测试只验证一个行为 |

## 代码审查（Google 原则）

### 审查标准

> **核心原则**：一旦代码确定能提升系统代码健康度，就应该批准，即使代码不完美。

详见 [Google Code Review Standard](https://google.github.io/eng-practices/review/reviewer/standard.html)

| 规则 | 说明 |
|------|------|
| **强制** | 技术事实和数据优先于个人偏好 |
| **强制** | 风格问题遵循 style guide |
| **强制** | 设计问题基于原则评估，非个人意见 |
| **强制** | 无明确规则时，与现有代码保持一致 |
| **推荐** | 寻求持续改进，而非完美 |

### 提交前检查

| 检查项 | 说明 |
|--------|------|
| 功能正确性 | 代码实现了预期功能 |
| 代码可读性 | 命名清晰、逻辑易读 |
| 错误处理 | 处理了异常情况 |
| 性能考虑 | 无明显性能问题 |
| 安全考虑 | 无安全漏洞 |
| 测试覆盖 | 有必要的单元测试 |
| 量化指标 | 符合函数≤30行、文件≤300行要求 |

## Git 提交规范

### 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| docs | 文档更新 |
| style | 格式调整（不影响代码） |
| refactor | 重构 |
| test | 测试相关 |
| chore | 构建/工具相关 |

### 提交规则

| 规则 | 说明 |
|------|------|
| **强制** | 主题不超过 50 字符 |
| **强制** | 主题小写开头，祈使句 |
| **强制** | 正文解释 what 和 why，不解释 how |
| **强制** | 每次提交原子化（单一变更） |
| **推荐** | 关联 Issue 编号 |
