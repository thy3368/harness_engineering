# 编码规范总则

## 概述

本规范为恒生电子股份有限公司编码规范的总则，规定了程序编写的基本要求。所有项目在软件开发和技术支持过程中必须严格遵守本规范。

详见 [编码规范总则](./编码规范总则.md)

## 基本要求

| 规则 | 说明 |
|------|------|
| **强制** | 应用程序应按照相关编码语言的标准风格书写 |
| **强制** | 程序结构清晰、易读，单个函数的行数应控制在 100 行以内 |
| **强制** | 不要随意定义全局变量，尽量使用局部变量 |
| **强制** | 变量命名应符合项目组的命名约定 |
| **强制** | 不准使用不符合项目组规定的开发工具 |
| **推荐** | 尽量使用标准库函数和公共函数 |
| **推荐** | 多使用括号以避免二义性 |

详见 [编码规范总则 - 基本要求](./编码规范总则.md#1-1-基本要求)

## 可读性要求

| 规则 | 说明 |
|------|------|
| **强制** | 所有命名中应使用标准的英文单词或缩写，名称应含义清晰 |
| **强制** | 每个源程序文件应有符合公司约定的文件头说明 |
| **强制** | 每个函数应有符合规定的函数头说明 |
| **强制** | 主要变量定义或引用时应加注释以反映其含义 |
| **强制** | 常量定义应加注相应说明 |
| **强制** | 处理过程的每个阶段应有相关注释说明 |
| **强制** | 技术支持阶段修改代码必须增加注释，指明时间、修改原因和修改人 |
| **推荐** | 保持注释与代码完全一致，边写程序边加注释 |
| **推荐** | 利用缩进来显示程序的逻辑结构，缩进量一致 |
| **推荐** | 循环、分支层次不宜太多，尽量保持在三层以内 |

详见 [编码规范总则 - 可读性要求](./编码规范总则.md#1-2-可读性要求)

## 结构化要求

| 规则 | 说明 |
|------|------|
| **强制** | 禁止出现两条等价的支路 |
| **强制** | 函数最好只有一个出口 |
| **推荐** | 用 IF 语句来强调只执行两组语句中的一组 |
| **推荐** | 用 CASE 实现多路分支，CASE 内部尽量少用分支语句 |
| **推荐** | 循环语句中避免多个出口 |
| **推荐** | 避免不必要的分支 |
| **推荐** | 不要轻易用条件分支去替换逻辑表达式 |

详见 [编码规范总则 - 结构化要求](./编码规范总则.md#1-3-结构化要求)

## 正确性与容错性要求

| 规则 | 说明 |
|------|------|
| **强制** | 程序首先是正确，其次才是优美 |
| **强制** | 编写完一段程序后应先回头检查 |
| **强制** | 充分考虑边界条件 |
| **强制** | 所有异常都需要被捕获并处理 |
| **推荐** | 进行必要的边界测试 |

详见 [编码规范总则 - 正确性与容错性要求](./编码规范总则.md#1-4-正确性与容错性要求)

## 可重用性要求

| 规则 | 说明 |
|------|------|
| **强制** | 相同功能的代码应抽象为公共函数或模块 |
| **强制** | 公共函数应具有通用性，不应包含特殊的处理逻辑 |
| **推荐** | 提取可重用的组件，提高代码复用率 |
| **推荐** | 遵循 DRY（Don't Repeat Yourself）原则 |

详见 [编码规范总则 - 可重用性要求](./编码规范总则.md#1-5-可重用性要求)

## 独立性要求

| 规则 | 说明 |
|------|------|
| **强制** | 模块之间应保持低耦合 |
| **强制** | 模块内部应保持高内聚 |
| **推荐** | 避免模块之间的直接依赖 |
| **推荐** | 使用接口或抽象类进行解耦 |

详见 [编码规范总则 - 独立性要求](./编码规范总则.md#1-6-独立性要求)

## 源文件版本及修订记录

| 规则 | 说明 |
|------|------|
| **强制** | 每个源程序文件头部必须有版本及修订记录描述 |
| **强制** | 修订记录应包含：日期、版本号、修改人、修改说明 |
| **推荐** | 使用版本控制系统管理代码变更 |

详见 [编码规范总则 - 源文件版本及修订记录描述要求](./编码规范总则.md#2-源文件版本及修订记录描述要求)
