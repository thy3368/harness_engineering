# 代码质量指南

## 概述

本指南参考业界开源项目（如 lifedever/claude-rules、Cursor Rules、CLAUDE.md）的最佳实践，为 AI 编码助手提供代码质量的核心原则和约束。

> **核心原则**：不要模仿遗留代码中的坏习惯，应按照标准重构代码。

## 量化指标

| 指标 | 限制 | 说明 |
|------|------|------|
| 单个函数 | ≤30 行 | 超过需拆分 |
| 单个文件 | ≤300 行 | 超过需拆分 |
| 函数参数 | ≤4 个 | 超过需封装为对象 |
| 嵌套层次 | ≤3 层 | 超过需重构 |
| 循环/分支 | ≤3 层 | 保持扁平 |

## 代码结构与组织

### 模块化设计

| 规则 | 说明 |
|------|------|
| **强制** | 采用逻辑清晰的代码组织，明确分离关注点 |
| **强制** | 使用函数和类将问题分解为更小、自包含的单元 |
| **强制** | 确保代码组件的模块化和可重用性 |
| **强制** | 遵循单一职责原则：每个函数/类应只负责一项特定任务 |
| **强制** | 禁止出现两条等价的支路 |
| **推荐** | 函数最好只有一个出口 |
| **推荐** | 处理复杂问题时，在编写代码前先制定高层次计划 |

### 遗留代码态度

| 规则 | 说明 |
|------|------|
| **核心** | 不模仿遗留代码中的坏习惯，应按照标准重构 |
| **强制** | 发现代码不符合规范时，应主动重构而非盲目遵循 |
| **强制** | 新代码必须使用现代 API 和最佳实践 |
| **推荐** | 如有时间，逐步改善遗留代码 |

### 数据结构与算法

| 规则 | 说明 |
|------|------|
| **强制** | 选择合适的数据结构和算法 |
| **推荐** | 注重清晰性与效率的平衡 |
| **推荐** | 优先考虑时间复杂度和空间复杂度 |

## 编码风格

### 缩进与格式

| 规则 | 说明 |
|------|------|
| **强制** | 使用 4 个空格保持一致的缩进（推荐使用空格而非制表符） |
| **强制** | 将代码行保持在合理长度（80-100 个字符），以提升可读性 |
| **强制** | 使用空行分隔不同的逻辑代码块，增强视觉组织性 |
| **强制** | 尾部保留一行空行 |

### 命名规范

| 类型 | 规则 | 示例 |
|------|------|------|
| 类名 | 名词或名词短语 | `UserService`, `OrderRepository` |
| 方法名 | 动词或动词短语 | `getUserById`, `calculateTotal` |
| 常量名 | 全大写下划线 | `MAX_RETRY_COUNT` |
| 布尔变量 | `is/has/can` 前缀 | `isValid`, `hasPermission` |
| 变量名 | 小驼峰，见名知意 | `userList`, `totalAmount` |

| 规则 | 说明 |
|------|------|
| **强制** | 使用有意义且描述性强的名称 |
| **强制** | 避免使用单字母或难以理解的缩写（循环计数器 `i`, `j` 除外） |
| **强制** | 同一项目内相同概念使用一致命名 |

### 注释与文档

| 规则 | 说明 |
|------|------|
| **强制** | 公共 API 必须有完整的文档注释 |
| **强制** | 注释对于多条采用 `/* */`，对于单条采用 `//` |
| **强制** | 技术支持阶段修改代码必须增加注释，指明时间、修改原因和修改人 |
| **推荐** | 保持注释与代码完全一致，边写程序边加注释 |
| **推荐** | 审慎使用注释，用于解释非显而易见的逻辑 |
| **推荐** | 避免对不言自明的代码过度注释 |

## 语言特定规则

### Java (Java 17+)

| 规则 | 说明 |
|------|------|
| **强制** | 优先使用 `record` 代替纯数据类 |
| **强制** | 使用 `sealed` 类限制继承层次 |
| **强制** | 使用 `Pattern Matching` 进行类型检查 |
| **强制** | 优先使用 `Optional` 处理空值 |
| **禁止** | 使用 `null` 返回值表示"无数据" |
| **禁止** | 使用 `String` 作为方法重载的唯一区别 |

### TypeScript

| 规则 | 说明 |
|------|------|
| **强制** | 禁止使用 `any` 类型 |
| **禁止** | 使用 `enum`，使用 `const` 对象代替 |
| **强制** | 使用 `import type` 导入类型 |
| **强制** | 使用 `as const` 声明常量对象 |
| **禁止** | 避免 barrel 文件（index.ts 重导出） |

### Python

| 规则 | 说明 |
|------|------|
| **强制** | 使用类型注解（Type Hints） |
| **强制** | 使用 `Protocol` 定义结构化类型 |
| **强制** | 使用 `ruff` 作为 linter |
| **推荐** | 使用 `uv` 或 `poetry` 管理依赖 |
| **禁止** | 使用 `from xxx import *` |

### Go

| 规则 | 说明 |
|------|------|
| **强制** | 使用错误包装（`fmt.Errorf` with `%w`） |
| **强制** | 使用小接口（不超过 3 个方法） |
| **强制** | 使用结构化并发（goroutine + channel） |
| **禁止** | 使用全局变量 |
| **推荐** | 优先使用切片而非数组 |

### Rust

| 规则 | 说明 |
|------|------|
| **强制** | 遵循所有权/借用规则 |
| **强制** | 使用 `thiserror` 或 `anyhow` 处理错误 |
| **强制** | 优先使用迭代器而非显式循环 |
| **强制** | 运行 `clippy` 并修复警告 |
| **禁止** | 使用 `unwrap()` 在生产代码中 |

### Vue 3

| 规则 | 说明 |
|------|------|
| **强制** | 使用 `<script setup>` 语法 |
| **强制** | 区分 `ref` 和 `reactive` 的使用场景 |
| **强制** | 使用 composable 函数封装复用逻辑 |
| **禁止** | 在 `setup` 中直接使用 async/await（使用 `suspense` 或 `useAsync`） |

### React

| 规则 | 说明 |
|------|------|
| **强制** | 遵循 Hooks 规则（只在顶层调用 Hooks） |
| **强制** | 正确使用 `useEffect`（包含 cleanup） |
| **强制** | 状态分层：UI 状态 vs 业务状态 vs 服务端状态 |
| **推荐** | 组件保持小而专注 |

## 编码最佳实践

### Clean Code 原则

| 规则 | 说明 |
|------|------|
| **强制** | 编写清晰且可读性强的代码 |
| **强制** | 避免重复代码（DRY 原则） |
| **强制** | 避免硬编码，使用常量或配置替代 |
| **推荐** | 从简单入手，随后再进行优化（YAGNI 原则） |
| **推荐** | 优先考虑代码结构和风格的明确性 |

### 错误处理

| 规则 | 说明 |
|------|------|
| **强制** | 异常信息应清晰描述错误原因和位置 |
| **强制** | 不捕获异常后不做任何处理 |
| **强制** | 关闭资源时使用 try-with-resources 或 defer |
| **推荐** | 自定义异常应继承适当的基类 |
| **推荐** | 区分可恢复异常和不可恢复异常 |

### 测试

| 规则 | 说明 |
|------|------|
| **强制** | 测试用例应覆盖正常情况、边界情况和异常情况 |
| **强制** | 单元测试必须通过才能提交 |
| **推荐** | 遵循测试金字塔原则 |
| **推荐** | 保持测试用例的独立性和可重复性 |

## 代码审查要点

### 提交前检查清单

| 检查项 | 说明 |
|--------|------|
| 功能正确性 | 代码是否实现了预期功能 |
| 代码可读性 | 命名是否清晰、逻辑是否易读 |
| 错误处理 | 是否处理了异常情况 |
| 性能考虑 | 是否有明显的性能问题 |
| 安全考虑 | 是否有安全漏洞（SQL 注入、XSS 等） |
| 测试覆盖 | 是否有必要的单元测试 |
| 量化指标 | 是否符合函数≤30行、文件≤300行等要求 |

### 审查反馈处理

| 规则 | 说明 |
|------|------|
| **强制** | 认真对待每条审查意见 |
| **强制** | 对于不认同的意见，应提供技术依据进行讨论 |
| **推荐** | 及时响应审查反馈，避免阻塞流程 |
